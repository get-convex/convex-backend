title: スキーマの考え方
sidebar_position: 450
description: "Convex におけるスキーマ設計の考え方とベストプラクティス"
---

Convex では、`CREATE TABLE` 文を書く必要も、フィールド名や型を決めるために
事前にテーブル構造をじっくり考える必要もありません。オブジェクトを Convex に
そのまま保存して、どんどんアプリを作っていけばよいのです。

ただし、最初に素早く進めすぎると、あとになって問題になることがあります。
「このフィールドって数値だっけ？文字列だっけ？あのバグを直したときに
変えた気がするけど？」

あまりに許容度の高いストレージシステムは、システムが成熟してきて、
自分のシステムの中にどんなデータが入っているかをきちんと把握したくなったときに、
負債になってしまうことがあります。

良い知らせとして、Convex は常に型付きです。ただし、その型付けは暗黙的です！
ドキュメントを Convex に送信すると、Convex はそのドキュメント内のすべての
フィールドの型を追跡します。[dashboard](/dashboard.md) にアクセスすれば、
任意のテーブルの推論されたスキーマを表示して、現状を把握できます。

「文字列から数値に変えたあのフィールドはどうなるの？」これも Convex が
処理できます。Convex はそういった変更も追跡し、この場合は
`v.union(v.number(), v.string())` のようなユニオン型として扱います。
こうして、ドキュメントのフィールドや型について後から考えを変えても、
Convex がきちんと対応してくれます。

スキーマをきちんと形式化する準備ができたら、
[schema builder](/database/schemas.mdx) を使ってスキーマを定義し、
スキーマバリデーションを有効にして、それに基づいた型を生成できます。
